Let's Be Technical

Nitty Gritty Details of Git

October 1, 2015


What is version control? What are the benefits of it?

Version control is a method of organizing, monitoring, and recording succesive changes made to a file (in our case, we'll talk about working with code). Let's say you're working on an application. You create a fresh file and start codin' away! You work on your app for a couple days, then go on a trip to Greece 'cus you've always wanted a pedicure where you stick your feet in a pool of fish. You come back with smooth, fish-nibbled feet, but wait, you have no idea what you did with your code. Did you add the Photoshopped logo? What about fixing the JS sign-in page? Now you're stressed and the skin on your feet start to crack. So much for a vacay.

WELL, there's still hope. What if the next time you made any change, that change was tracked? Then as you make more and more changes, the progression of your program is suddenly clear and you can SEE those changes. Sounds divine, right? It is thanks to version control. As you continue on your app, you install version control software on your computer. Now as you write code, you sync your changes to the software and BADA-BOOM you've got a beautifully compiled list of all your changes.

Now let's say your app is building traction. You tell your grandmother about it, whose dentures specialist's secretary's cousin happens to hear about it and next thing you know you've got a team working on your idea. Now version control becomes even more essential. You have multiple people working on the same piece of code, so you're going to want to keep constant track of what's being edited, removed, and added and who's doing what. With version control, you can be a majestic hawk overlooking how your code evolves.


How does git help you keep track of changes?

Lemme explain what exactly this 'version control software' is. Git is the name of this special software and it's the mystical (it's not actually mystical but you'll FEEL mystical using it) power behind all those changes you've been tracking. It does the tracking FOR YOU.

Let's say you're back at it with that Javascript sign-in page. You fer sure want git to know about any edits to this part of the code. So, you open up good 'ol Terminal. This is where all the git magic happens. Through a system called branching, you seperate from the master version of your code with 'branches' or copies of the master version. It's best that you never work directly off your master version because, well it's the master. If you screw up the master, an actual master of a Scottish manor will float to your house and reprimand you, IN PERSON. You won't get physically hurt, but the shame will be there. So don't let that happen. Work from a branch. Now that you have created a new branch from the master by typing 'git checkout -b branch_name' you can work with the code where the user's sign-up information is stored. Perfecto, you're on the right track towards letting git know you've made a change.

Next, you type 'git status' and see that the latest edit (that JS signup page) has not been 'staged.' Staging is the step where you get your file camera-ready for what's called a commit. You must stage your file before 'committing' to a change. To stage a change you simply type 'git add [FILE NAME]'. Now you can proceed to committing the change so git can track it. You type 'git commit -m "commit message." Now don't make a lousy commit message like "sign up page". 'Sign-up page' is lousy for a couple reasons. First, it must include a verb, specifically in the imperative, present tense. So you'd need to edit that message to "Add sign-up page." Secondly, you'll want to be specific, but keeping the number of characters to between 50 and 72. Why? Read about it here. A re-edit of the commit message would result in "Grab user value and store in variable." See the difference? So specific! Now that code edit is recorded and you, along with your team members, can refer to it whenever, even after another vacation to Cancun 'cus you heard Kim Kardashian is going to be there and you want to witness botox injections gone wrong. No wonder she wears long duster coats more nowadays.


Why use GitHub to store your code?

Thanks to your grandma's denture specialist's secretary's cousin's help of recruiting more zen developers like yourself to work on your app, you've got a team. There's just one problem. Everyone is located in a different country. You know that the Internet is how you can share each other's code, but what about git? Everyone only has git on their own, local computers. Well some people thought of that and sorted out the logisitcs for you by creating a website called GitHub. GitHub is the pretty online coordinator with git that houses all your code in one place, that's accessible to anyone on your team, and allows for changes to be tracked chronologically and visually. Yes you can actually compare the old and new changes to your code side by side. It's suh great.
