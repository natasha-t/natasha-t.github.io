SQL Optimization

Cuz There's So Much Freakin' Data to Sift Through

November 20, 2015


Optimizing SQL

SQL, or Structured Query Language, is a language used to communicate with databases so you can pull the information you want from them. The langauge reads much like English, using words like SELECT and FROM to select what you want to select and from what table. I'm fairly new to this realm of databases so I don't know much, but apparently the language could use some optimization in order to make query results more efficiently.

Databases, consisting of tables, can have millions of rows of information. Using commands like SELECT row_name, row_name FROM table_name ORDER BY some_id ASC will get the job done as far as pulling the data you request, but when you're dealing with millions of table rows, how effective is using the above mentioned command? Do you really have the time and energy to sift through hundreds of rows? I think not. This is where optimization can come in. After doing some research, I found some tips on how to optimize SQL searches. Here are a few (source citations are at the end of the post): (this list is a work in progress)


Understanding Indexes

Apparently indexes in tables are commonly misunderstood among SQL noobs (like me). Index values in a database table are simply non-significant numerical values that serve as counters for rows. Indexes are significant in that they provide structure to the table itself as opposed to changing the table data. It's common that beginners index everything, without understanding that doing so will make it harder to insert, delete, and update rows later. The way databases handle long lists of indexes is via doubly linked lists or leaf nodes.

A doubly linked list organizes the index in a logical way, as opposed to in a physical way. If an insertion needs to be made, a new node is created between pre-exisiting nodes, with the pre-existing nodes including links to the new node. This method of connecting new insertions to pre-existing ones via links results in a tree like structure and most importantly, no obstruction to the index values. The problem with this leaf node structure is that the computer must traverse all nodes when searching for an index, resulting in slow serach times. Since I just learned about how databases are structured like, literally two days ago, this all seems rather abstract to me. But it's good to know that there are ways in which databases deal with the hardline ascending numerical structure of indexes.


Symbol Operators

Using the '*' symbol will pull all columns from a table that fit a certain condition defined later in the command. But sometimes you can pull columns you don't need so it's best to specify the exact column names after SELECT that you wish to pull data from. Also, the LIMIT clause may come in handy when you don't want rows on rows on rows getting spat back at you; LIMIT will reduce the amount of rows being pulled to whatever you specify.

Another way to shorten search time is to provide more precise conditions for the number of columns requested to be pulled. For example, to search for a values up to 10, it'd be more efficient to query >= 10, as opposed to > 11, in which case the computer would jump to value 11 then back to 10.

It is best to avoid using the NOT operator because not only will the correct data be sifted through and pulled, all the information that is not what you want pulled will be searched too, thus slowing query time.
